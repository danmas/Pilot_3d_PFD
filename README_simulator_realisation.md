# Реализация симулятора полёта — обзор и анализ

## ✅ Оценка архитектуры — Отлично

Реализация **Варианта А (серверный симулятор)** следует плану из `analysis_results.md`. Целостность потоков данных сохранена, а для анализа модели добавлен отдельный blackbox-контур:

```
┌─────────────┐     ┌──────────────────┐     ┌───────────┐
│  UDP 14443  │──▶  │  bridge-plugin   │──▶  │ SSE / PFD │
│ (реальные   │     │  publishDecoded  │     │ React App │
│  данные)    │     │  Frame()         │     └───────────┘
└─────────────┘     │        ▲         │
                    │   ┌────┴─────┐   │
┌─────────────┐     │   │ Capture  │   │
│ FlightSim   │────▶│   │ .jsonl   │   │
│ .step(dt)   │     │   │          │   │
└─────────────┘     └───┴──────────┴───┘
       │
       └──▶ sim-blackbox.v1 (*.jsonl)
```

И реальные UDP-пакеты, и кадры симулятора проходят через **один и тот же** конвейер `publishDecodedFrame()` / `applyDecFormulas()` / SSE-вещание. Это означает, что все потребители (PFD, Diagnostic Viewer, Capture) работают одинаково в обоих режимах. Для симулятора дополнительно пишется `sim-blackbox.v1`: raw-действия пилота, сглаженные controls, состояние FDM и промежуточная физика (`Cl`, `Cd`, `lift`, `drag`, `thrust`, ускорения, rate-команды).

## Модель симулятора (`simulator.ts`) — Добротно

Модель динамики полета правильно подобрана для визуализационного симулятора:

| Характеристика | Реализация | Оценка |
|---|---|---|
| **Начальный режим** | По умолчанию 250 узлов, 10000 футов, РУД 0.6, pitch 3° | `trim_hold_60s` держит CAS/высоту/G почти без дрейфа |
| **Приёмистость двигателя** | Апериодическое звено 1-го порядка, τ ≈ 0.4 с | Реалистичное поведение |
| **Атмосфера** | Модель плотности по ISA | Хорошо для учёта высоты |
| **CAS/TAS** | Внутренний `tasMs`, CAS вычисляется для PFD | Убрана просадка скорости на первом tick |
| **Trim lift** | `trimCl` и `trimAlphaRad` пересчитываются на `reset()` | Начальный режим согласован по lift |
| **Тангаж** | Сглаженная pitch-rate команда от ручки | Кинематическая модель, не моментная динамика |
| **Автоцентрирование крена** | `rollDamping` при отпущенной ручке | Естественное поведение |
| **Столкновение с землёй** | Ограничение высоты с гашением энергии | Хорошая страховка |

Метод `step()` возвращает ровно тот плоский словарь `Record<string, number | null>`, который ожидает `publishDecodedFrame()` — поэтому `applyDecFormulas()` обогащает его идентично реальным UDP-кадрам.

Подробная физика, ограничения и формулы вынесены в `README_flight_physics.md`.

## Интеграция в фронтенд (`App.tsx`) — Полноценная

Представление PFD теперь включает:

1. **Переключатель UDP / Симулятор** — `POST /api/simulator/mode` переключает между реальными данными и симуляцией
2. **Клавиатурное управление** на 20 Гц — WASD/QE/Shift/Ctrl с плавными пружинами центрирования (`* 0.35` за тик), плюс raw pilot snapshot для blackbox
3. **Визуальный индикатор отклонения ручки** — показывает текущее положение по крену/тангажу из `FCU_Roll_Left`/`FCU_Pitch_Left`
4. **Индикатор РУД** — прогресс-бар из `Engine_N1_Target_Left`
5. **Управление записью** — Старт/Стоп захвата, список последних записей, воспроизведение
6. **Scripted Profiles** — offline-прогоны профилей с выбором initial conditions
7. **Таймлайн воспроизведения** — ползунок для пролистывания `.jsonl` прямо со страницы PFD

## Blackbox и профили для аналитиков

Для ручного полёта симулятор пишет пару файлов:

- `*_live.jsonl` — обычный `telemetry-frame.v1`, то, что видит PFD/replay.
- `*_sim_blackbox.jsonl` — `sim-blackbox.v1`, причинная запись "пилот → controls → physics → state".

Для воспроизводимых проверок добавлен запуск scripted profiles из UI (`Live` → `Scripted Profiles`) и через API. При запуске из UI созданный telemetry-файл сразу открывается как Replay: приборы сбрасываются на первый кадр профиля и проигрывают весь сценарий от выбранных initial conditions.

- `GET /api/simulator/profiles`
- `GET /api/simulator/profile-presets`
- `POST /api/simulator/profile/run`

Профили:

| Профиль | Назначение |
|---|---|
| `trim_hold_60s` | Нейтральные controls, проверка trim/drift |
| `pitch_step_up` | 5 секунд pitch `+1`, затем neutral |
| `pitch_step_down` | 5 секунд pitch `-1`, затем neutral |
| `roll_command_step` | 3 секунды full-right roll command, затем neutral |
| `throttle_step` | throttle `0.6 -> 1.0` |
| `combined_maneuver` | Комбинированный pitch/roll/rudder/throttle сценарий |

Initial presets:

| Preset | Условия |
|---|---|
| `cruise_10000_250` | `10000 ft`, `250 kt`, `throttle 0.6`, `pitch 3°` |
| `low_speed_3000_160` | `3000 ft`, `160 kt`, `throttle 0.55`, `pitch 5°` |
| `high_altitude_25000_250` | `25000 ft`, `250 kt`, `throttle 0.72`, `pitch 4°` |
| `approach_1500_140` | `1500 ft`, `140 kt`, `throttle 0.5`, `pitch 4°` |

Пример:

```powershell
Invoke-RestMethod -Method Post `
  -Uri http://localhost:3410/api/simulator/profile/run `
  -ContentType "application/json" `
  -Body '{"profileId":"pitch_step_up","presetId":"cruise_10000_250"}'
```

## Найденные проблемы

### 1. ⚠️ `VerticalSpeed.tsx` — рассогласование масштаба Vy

В `VerticalSpeed.tsx` (строка 9) `Vy` используется напрямую:

```ts
const vs = (frame.Vy as number) ?? 0;
// ...
<line ... y1={getVsY(vs/1000)} ... />
```

Деление на 1000 предполагает, что `Vy` — в **мм/с**. В `field-catalog.ts` (строки 317–323) комментарий к `Vy` — «Вертикальная скорость барометрическая» — вероятно, уже в **м/с** из tnparserrt. В симуляторе (строка 198) значение задаётся как `this.vy * 1000` (мм/с), что соответствует ожиданию компонента. Но если реальный UDP `Vy` приходит в м/с, индикатор вертикальной скорости будет показывать в 1000 раз меньше в режиме реального потока. **Проверь фактический масштаб `Vy` из tnparserrt.**

### 2. ⚠️ `simulator.ts` — `payload.Time` использует `Date.now()`

В `simulator.ts` (строка 200):

```ts
payload.Time = Date.now();
```

Это записывает в поле `Time` (ARINC param "0150") миллисекунды Unix-эпохи. В реальных UDP-данных это поле, вероятно, представляет относительное или абсолютное полётное время, а не настенные часы. Если какой-либо потребитель интерпретирует `Time` как значимый телеметрический параметр, в режиме симулятора он получит некорректные данные. Сейчас ни один PFD-компонент не использует `Time`, но стоит иметь это в виду.

### 3. ⚠️ Обработчик клавиш: `Space` сбрасывает, и `R` тоже сбрасывает

В `App.tsx` (строки 247–249):

```ts
if (keys.has(' ') || keys.has('r')) {
    resetSimulation();
}
```

Пробел обычно используется для прокрутки страницы. Хотя вызывается `e.preventDefault()` для пробела, если поле ввода было в фокусе, а затем фокус ушёл во время полёта, нажатие пробела может неожиданно сбросить симуляцию. Клавиша `R` также сбрасывает — кратковременное нажатие `R` при наборе «Reset» где-либо может вызвать сброс. Рассмотри менее конфликтную клавишу, например **Backspace**, или потребуй **двойное нажатие**.

### 4. ⚠️ `AltitudeTape.tsx` — семантическое несоответствие `StandardAltitude`

В `AltitudeTape.tsx` (строка 15):

```ts
const selectedAltFt = finiteNumber(frame.StandardAltitude);
```

Подсказка гласит «Заданная высота автопилота (Selected Altitude)». Но используется поле `StandardAltitude` (param "0203", комментарий «Стандартная высота»), а не параметр «заданной высоты». Это может быть осознанным решением (отдельного поля заданной высоты в каталоге нет), но именование вводит в заблуждение.

### 5. ℹ️ Пружины центрирования рассинхронизируются после сброса

В `App.tsx` (строки 200–203) локальные переменные `currentRoll`/`currentPitch`/`currentRudder`/`currentThrottle` хранят сглаженное положение ручки. При вызове `resetSimulation()` через клавиатуру **сервер** сбрасывается, но **клиентские** переменные сохраняют последние значения. Они сойдутся к центру за несколько тиков благодаря пружине центрирования, но будет краткий момент рассинхронизации. Мелкий косметический дефект.

## Предложения

1. **Добавить формулу `dec_VyMs`** — если `Vy` из tnparserrt действительно в мм/с, добавить `dec_VyMs: Vy / 1000` в расчётные формулы, чтобы инструменты использовали согласованную единицу измерения.

2. **Поддержка геймпада** — в `analysis_results.md` упоминается Gamepad API. Добавление лёгкого опроса `navigator.getGamepads()` в клавиатурный интервал было бы естественным следующим шагом.

3. **Настройка частоты симулятора** — 25 Гц (40 мс) зашито жёстко. Можно сделать настраиваемым через `.env` или API.

4. **Регулировка балансировки симулятора** — разрешить тонкую настройку балансировочного режима (сейчас все константы `readonly`) через API, чтобы можно было тестировать разные конфигурации самолёта.

## Вердикт

Реализация **чистая, хорошо структурированная и готовая к использованию**. Ключевой принцип — проведение кадров симулятора через тот же конвейер, что и реальные UDP-кадры — корректно выполнен повсюду. Модель динамики полета уместно упрощена для визуализации приборов без излишнего упрощенчества. Элементы управления во фронтенде продуманы (пружины центрирования, визуальный индикатор ручки, подсказки клавиш). Найденные проблемы незначительны и ни одна не является блокирующей. Отличная работа.

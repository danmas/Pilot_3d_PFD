# 3D Aircraft Instrument

Трёхмерный прибор, отображающий самолёт в пространстве: процедурная модель или загружаемая `.glb`-модель в небесной сфере с линией горизонта и числовым HUD. Камера орбитирует вокруг самолёта; доступны пресеты видов, кнопочное вращение и смена модели с сохранением параметров.

Зарегистрирован в PanelKit как `aircraft-3d` (иконка `Plane`) и доступен для добавления через PanelBuilder.

## Загрузка и смена моделей

### Манифест `models.json`

Файл `public/data/aircraft3d/models/models.json` описывает доступные GLB-модели:

```json
{
  "models": [
    {
      "id": "tu204",
      "label": "Ту-204",
      "url": "/data/aircraft3d/models/tupolev_tu_204.glb",
      "scale": 1.0,
      "yawOffsetDeg": 180,
      "offsetX": 7,
      "offsetY": 0,
      "offsetZ": -10
    }
  ]
}
```

### Параметры модели (ModelEntry)

| Параметр | Тип | По умолчанию | Назначение |
|---|---|---|---|
| `id` | `string` | — | Уникальный идентификатор |
| `label` | `string` | — | Отображаемое имя в селекторе |
| `url` | `string \| null` | `null` | URL `.glb` файла (`null` = примитивы) |
| `scale` | `number` | `1.0` | Множитель масштаба поверх авто-фита |
| `yawOffsetDeg` | `number` | `0` | Поворот модели вокруг Y (корректировка «носа») |
| `offsetX` | `number` | `0` | Сдвиг модели по X (влево/вправо) |
| `offsetY` | `number` | `0` | Сдвиг модели по Y (вверх/вниз) |
| `offsetZ` | `number` | `0` | Сдвиг модели по Z (вперёд/назад) |

### Загрузка (`modelConfig.ts`)

Функция `fetchModels()` загружает манифест при монтировании прибора:
- `GET /data/aircraft3d/models/models.json`
- «Примитивная» модель добавляется первой автоматически
- При ошибке/отсутствии файла возвращается только примитивная

### Авто-фит масштаба (AircraftModel.tsx → GLBAircraft)

При загрузке `.glb`:
1. Вычисляется bounding box модели (`Box3.setFromObject`)
2. Находится максимальный габарит (X, Y или Z)
3. Модель масштабируется так, чтобы максимальный габарит = `TARGET_SIZE` (8 единиц)
4. Модель центрируется (сдвиг на `-center`)
5. Параметр `scale` применяется поверх авто-фита

### Сохранение (API)

```
PUT /api/aircraft3d/models
Body: { "models": [ ...GLB модели... ] }
```

Обработчик в `bridge-plugin.ts`:
- Принимает массив моделей (примитивная исключается — она всегда добавляется автоматически)
- Записывает в `public/data/aircraft3d/models/models.json`
- После сохранения `fetchModels()` перечитывает манифест

## Алгоритмы

### Ориентация самолёта (AircraftModel)

Телеметрия → углы Эйлера → плавная интерполяция:

```
TelemetryFrame.PitchAngle  →  rotation.x  (тангаж)
TelemetryFrame.RollAngle   →  rotation.z  (крен)
TelemetryFrame.Heading1    → -rotation.y  (курс, инвертирован)
```

Порядок вращения: **YXZ** (yaw → pitch → roll) — стандартная авиационная конвенция.

Каждый кадр (`useFrame`) текущее вращение `group.rotation` линейно интерполируется (lerp, фактор 0.12) к целевому. Это сглаживает скачки между кадрами телеметрии (~25 Гц) до плавного 60 FPS движения.

Сдвиги `offsetX/Y/Z` применяются как `position` на корневом `<group>` модели.

### Небесная сфера (HorizonSphere)

Сфера радиусом 200 единиц, рендерится изнутри (`side: BackSide`). GLSL-шейдер:

- **Верх** (y > 0): градиент от бледно-голубого у горизонта до тёмно-синего в зените
- **Низ** (y < 0): градиент от бледно-коричневого у горизонта до тёмно-коричневого внизу
- **Линия горизонта**: яркая полоса (±0.025 по y), белёсого цвета

Сфера привязана к мировым координатам — не вращается вместе с самолётом.

### Управление камерой (CameraController)

Двухуровневая система:

1. **OrbitControls** (мышь): свободное вращение, зум колесом, damping 0.08
2. **Imperative API** (кнопки): `setPreset()`, `rotateBy()`, `reset()` через `forwardRef` + `useImperativeHandle`

Плавный переход: `camera.position.lerp(target, 0.07)` каждый кадр.

### Позиционирование мира: WorldGroup и синхронизация наземных объектов

3D-сцена (RealAircraft3DScene.tsx) использует единый механизм для перемещения мира относительно самолёта:

```
                        HorizonSphere (мировые, статичны)
                        
                   ┌─────────────────────────────┐
                   │         WorldGroup (-ay)     │
                   │  ┌───────────────────────┐  │
                   │  │ GroundDisc  (Y=-6)    │  │
                   │  │ Runway      (Y=-5.97) │  │
                   │  │ Trees       (Y=-6)    │  │
                   │  │ RedTree     (Y=-6+7.5)│  │
                   │  │ Clouds      (Y=20-75) │  │
                   │  └───────────────────────┘  │
                   └─────────────────────────────┘
                        
                   AircraftModel (начало координат)
```

**WorldGroup** — корневая `<group>`, которая каждый кадр устанавливает свою позицию в `-aircraftPosition`. Это сдвигает весь «мир» в противоположную сторону от движения самолёта, создавая иллюзию полёта. Самолёт (AircraftModel) находится снаружи WorldGroup, в начале координат, и не двигается — камера всегда орбитирует вокруг него.

**Все наземные объекты находятся внутри WorldGroup** и их локальные Y-координаты фиксированы относительно земли (Y=-6). WorldGroup автоматически уводит их вниз при наборе высоты — синхронно, с одинаковой скоростью.

**GroundDisc** (зелёный круглый диск земли) — единственный объект, который **снаружи WorldGroup**. Он самостоятельно отслеживает `aircraftPosition` в каждом кадре:

```
g.position.x = aircraftPosition.x;   // следует за самолётом по X
g.position.z = aircraftPosition.z;   // следует за самолётом по Z
targetY = -6 - aircraftPosition.y;   // опускается синхронно с WorldGroup
g.position.y += (targetY - g.position.y) * 0.06;  // плавная интерполяция
```

То есть GroundDisc всегда находится на той же высоте, что и объекты внутри WorldGroup.

#### Почему важно не компенсировать WorldGroup обратно

Каждый объект внутри WorldGroup уже сдвинут на `-aircraftPosition.y`. Если объект **дополнительно** добавляет `-ay` в свой локальный Y (чтобы «скомпенсировать» WorldGroup), он получает двойное смещение `-2*ay` — и при наборе высоты уходит вниз вдвое быстрее, проваливаясь относительно земли.

**Правило:** Объекты внутри WorldGroup НЕ должны компенсировать `aircraftPosition.y`. Они стоят статично на своей высоте (Y=-6, -5.97 и т.д.). WorldGroup сам двигает их синхронно.

#### Схема Y-координат на земле (aircraftPosition.y = 0)

```
Y=+∞           ↑ небо
     ...
Y=20..75       □□□ облака (Clouds)
     ...
Y=-5.97        ██████████  ВПП (Runway)
Y=-6.00        ●●●●●●●●●●  земля (GroundDisc)
Y=-6.00        ♣♣♣♣♣♣♣♣♣♣  деревья (Trees — основание ствола)
Y=-6.00+7.5    ▲ красная ёлка — вершина конуса
```

При наборе высоты (aircraftPosition.y растёт) все Y-координаты уменьшаются на ту же величину — мир «опускается», и все объекты смещаются синхронно, без отрыва друг от друга.

### Историческая справка (v2.7.4 – v2.8.1)

**v2.7.4 — Двойная Y-компенсация + несинхронная земля**
До v2.7.4 GroundDisc компенсировал высоту по формуле `-6 - alt/500` (где `alt` — радиовысота из телеметрии), а объекты внутри WorldGroup компенсировали `aircraftPosition.y`. Эти две величины росли с разной скоростью, из-за чего при наборе высоты земля, ВПП и деревья расходились по высоте, создавая эффект «погружения» объектов под землю.

Исправление:
1. **GroundDisc** переведён на `aircraftPosition.y` (вместо `alt/500`) — синхронизирован с механизмом WorldGroup
2. **Runway, Trees, RedTree** — удалена избыточная компенсация `-ay` из локальных Y-координат (устранено двойное смещение `-2*ay`)

**v2.8.0 — Improved FDM: движение по XZ + исправление направления**
В Improved FDM (`tickImprovedFdm`) полностью отсутствовало движение `aircraftPosition` по X и Z — самолёт висел на месте, менялась только высота. Добавлено движение по курсу:

```
headingRad = -state.heading * DEG     // инверсия — конвенция проекта
forwardHoriz = cos(pitchAngle)
speedWU = speed * 0.5144 / 40         // knots → world-units/s
x += -sin(headingRad) * speedWU * forwardHoriz * dt
z += -cos(headingRad) * speedWU * forwardHoriz * dt
```

Направление движения согласовано с Simple FDM: инверсия курса (`-state.heading`), чтобы самолёт летел носом по курсу.

**v2.8.1 — Throttle: прямой тап на индикаторе (вместо rate-управления или дельт)**
Красный джойстик теперь управляет только рысканием (yaw) по горизонтали. Вертикальная компонента джойстика отключена.

Положение РУД (0–100%) устанавливается тапом или свайпом непосредственно по красному столбику-индикатору. Синий джойстик (крен/тангаж) без изменений.

## Архитектурные решения

- **`useGLTF`** из `@react-three/drei` для загрузки `.glb` с автоматическим кэшированием
- **Клонирование сцены**: `scene.clone(true)` — `useGLTF` кэширует оригинал, клон можно трансформировать
- **`<primitive>` вместо `<line>`**: JSX `<line>` конфликтует с SVG `<line>` в TypeScript
- **`/// <reference types="@react-three/fiber" />`**: каждый файл с Three.js JSX начинается с этой директивы
- **Suspense**: `<Canvas>` обёрнут в `<Suspense>` — Three.js и GLB загружаются асинхронно
- **Lerp**: камера (0.07) и модель (0.12) — плавное движение при ~25 Гц телеметрии

# Декодирование телеметрии (decoding.ts)

Главный документ, описывающий архитектуру, контракты, соглашения и сценарии устранения проблем в новой схеме приёма и декодирования данных из `tnparserrt.exe`.

---

## 1. Архитектура: два независимых источника

```
┌──────────────┐          ┌──────────────────┐
│   out.json   │          │ field-catalog.ts │
│   (слоты,    │          │   (имена,        │
│    порядок   │          │    типы,         │
│    байт)     │          │    описания)     │
└──────┬───────┘          └────────┬─────────┘
       │                           │
       │    ┌──────────────────────┘
       │    │  Сопоставление по ARINC param
       │    │  (двухпроходное)
       ▼    ▼
┌─────────────────┐
│  decoding.ts    │
│  buildDecode    │
│  Schema()       │
└────────┬────────┘
         │  DecodeSchema
         ▼
┌─────────────────┐
│  decodePayload()│
│  → Record<key,  │
│    number|null> │
└────────┬────────┘
         │  applyDecFormulas()
         ▼
┌─────────────────┐
│  TelemetryFrame │  → SSE, capture, instruments
│  (плоский       │
│   словарь)      │
└─────────────────┘
```

### out.json — источник байтовой раскладки
- Определяет КАКИЕ поля (слоты) передаются tnparserrt по данному UDP-порту.
- Определяет ПОРЯДОК этих полей в бинарном буфере.
- **Вставка/удаление слота изменяет байтовую раскладку** — это главная уязвимость.
- Редактируется вручную («кожаными мешками»).

### field-catalog.ts — источник имён и типов
- Содержит 132 записи: имя, тип данных, ARINC param, комментарий.
- **НЕ определяет порядок декодирования** (в отличие от старой схемы).
- Сопоставляется со слотами out.json по `param` (метка ARINC 429).
- Добавление/удаление записей в каталоге **не ломает декодирование**.

---

## 2. Двухпроходное сопоставление

`buildDecodeSchema()` выполняет два прохода:

1. **Проход по ARINC param**: слоты с `param` ищутся в каталоге.
2. **Позиционный проход**: оставшиеся слоты (без param — статусы, температуры блоков) сопоставляются с оставшимися записями каталога по индексу.

**Результаты валидации при старте:**
- Слот без соответствия → автоимя `slot_XXXX_auto` + WARNING.
- Запись каталога без слота → поле всегда `null` + WARNING.
- Повторное использование `param` → WARNING.

## 3. Конвенция dec_ префикса

| Префикс | Значение |
|---------|----------|
| Без префикса | Значение из бинарного потока tnparserrt (исходные байты поля) |
| `dec_` | Расчётное поле, вычислено функцией `applyDecFormulas()` в decoding.ts |

**Текущие расчётные поля:**

| Ключ | Формула | Назначение |
|------|---------|------------|
| `dec_BaroAltFt` | `BaroAltitude` | Барометрическая высота в футах |
| `dec_RadioAltFt` | `RadioAltitude` | Радиовысота в футах |
| `dec_MachKnots` | `MachNumber × 661.5` | Скорость в узлах (приблизительно) |
| `dec_G` | `NormalG` (алиас) | Перегрузка |

**Добавление нового расчётного поля:**
1. Внести функцию в `DEC_FORMULAS` в `decoding.ts`.
2. Документировать в этой таблице.

## 4. Порт по умолчанию: 14443

| Порт | Назначение | Потребитель |
|------|-----------|-------------|
| 14442 | Полный поток (исходный) | FTI.Monitor (WPF) |
| **14443** | **Полный поток (зеркало)** — **ПО УМОЛЧАНИЮ** | **Pilot_3d_PFD bridge** |
| 14444 | Компактный AVIONICS (9 полей) | Устарел, не используется |

Настройка: `--udpPort` флаг CLI или `BridgeOptions.udpPort`.

## 5. Формат данных

### Бинарный (UDP от tnparserrt)
- Маленькие фреймы (≤ MTU): один пакет = один фрейм.
- Большие фреймы: marker-пакет (sync 0x544E = "TN") + N пакетов данных.
- Сборка: `Buffer.concat(chunks)` → передаётся в `decodePayload()`.

### TelemetryFrame (JSON/SSE/Capture)
```typescript
{
  "schema": "telemetry-frame.v1",
  "seq": 42,
  "timeMs": 12345,
  "replayTimeMs": null,
  "receivedAt": "2026-05-21T12:00:00.000Z",
  "source": "tnparser-udp-14443",
  "PitchAngle": 3.14,          // прямое значение из парсера
  "RollAngle": -0.87,
  "CAS": 145.5,
  "RadioAltitude": 320.0,
  "dec_BaroAltFt": 1050.0,     // расчётное
  // ... все 132 поля + dec_*
}
```

## 6. Структура файлов

| Файл | Роль |
|------|------|
| `out.json` | Конфигурация tnparserrt: слоты по портам. **Единственный источник порядка байт.** |
| `field-catalog.ts` | Метаданные: имена, типы, ARINC param, описания. **Справочник, НЕ раскладка.** |
| `decoding.ts` | Загрузка out.json, сопоставление, декодирование, `dec_` формулы, валидация. |
| `bridge-plugin.ts` | Vite-плагин: UDP-сокет, сборка фреймов, SSE, capture, replay API. |
| `src/types.ts` | `TelemetryFrame` и `PFDFrame` (deprecated) типы для фронтенда. |

## 7. Сценарии: человек накосячил

| Ситуация | Поведение |
|----------|-----------|
| В out.json добавили слот в середину | Порядок перестроен авто, новое поле с автоименем `slot_XXXX_auto` |
| В out.json удалили слот | Поле получает `null` в словаре, WARNING при старте |
| В out.json изменили `param` | Старое имя теряет слот, новое получает автоимя. Проверь консоль! |
| В field-catalog.ts перепутан порядок записей | **Не влияет** — сопоставление по `param`, а не по индексу |
| В field-catalog.ts опечатка в `param` | Поле не найдено → `null` + WARNING |
| ARINC param дублируется в out.json | Второй дубликат получает автоимя, WARNING (используется первое совпадение) |

## 8. Диагностика при запуске

При старте `bridge-plugin` выводит в консоль:

```
[BRIDGE] Loading decode schema from out.json...
[BRIDGE] Found 132 slots for port 14443
[BRIDGE] Schema validation report:
Decode schema: 132 fields, 474 bytes/frame
Warnings: 0

Fields from catalog: 132
Auto-named fields:  0
Total frame size:   474 bytes
```

**Если есть WARNINGS** — изучи их немедленно. Каждый warning означает потенциально некорректные данные.

## 9. Обратная совместимость

- **Старые capture-файлы** (flight-frame.v1, pfd-frame.v1): читаются через `readJsonLines()` с проверкой схемы.
- **Новые capture-файлы**: `telemetry-frame.v1`.
- **Старые инструменты** (использующие `PFDFrame`): продолжают работать через те же SSE-потоки, но получат плоский словарь вместо вложенного. Требуется миграция.

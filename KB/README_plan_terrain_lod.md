## 1. Проблема

Реальный ландшафт (RealTerrainMesh) — один из самых тяжёлых компонентов сцены по количеству геометрии.

### Текущие характеристики (на момент 2026-06-14)

В [RealTerrainMesh.tsx](../src/components/Instruments/aircraft3d/terrain/RealTerrainMesh.tsx):
- `segX = Math.min(512, 64)`
- `segZ = Math.min(512, 256)`
- Вершин на тайл: `(64+1) × (256+1) ≈ 16 705`
- Треугольников на тайл: `64 × 256 × 2 = 32 768`

При типичной загрузке:
- 5×5 (25 тайлов) → **~820 тысяч** треугольников
- 7×7 (49 тайлов, текущий `loadRadius=3` в TerrainManager) → **~1.6 миллиона** треугольников **только на terrain**

Дополнительные проблемы:
- `computeVertexNormals()` на каждый батч обновления тайлов (CPU).
- `frustumCulled = false` на всех terrain-мешах.
- Геометрия пересоздаётся в `useMemo` при каждом изменении `tiles`.
- Семплинг высот — nearest-neighbour (`Math.round`).
- Нет никакой адаптивности: все тайлы одинаково детальные независимо от расстояния до камеры.

### Контекст сцены
- Камера обычно на высоте **4–22 WU** над землёй (chase `[0,4,12]`, top `[0,16,0.5]`), иногда до ~40 WU при орбите/зуме.
- Высоты сильно сжаты: `(h - globalMinElev) / 40` (1 WU ≈ 40 метров).
- Самолёт всегда в (0,0,0), мир движется через `WorldGroup` + `aircraftPosition`.
- Два режима отображения: `realistic` (спутниковая текстура) и `schematic` (wireframe).

При такой высоте и сжатии по вертикали высокая плотность вершин на дальних тайлах в значительной степени тратится впустую.

**Уже доступные данные для оптимизации** (из кода):
- В `TerrainTileData` хранятся `minElevation` и `maxElevation` (метры) на тайл.
- В `RealTerrainMesh` считается `globalMinElev` по текущей сетке тайлов (для бесшовности).

## 2. Рассмотренные варианты

### Вариант 1: Простое снижение segments (64×256 → 32×64 / 16×32)
**Плюсы:** максимальная простота, мгновенный эффект.  
**Точные цифры:** 32×64 → 4 096 треугольников на тайл (**8× снижение**). При 25 тайлах: ~820k → **~102k** треугольников.  
**Минусы:** равномерная потеря детализации. На близких расстояниях (chase, низкие пролёты) может выглядеть блочно.  
**Оценка:** Отличный первый шаг. Для chase-камеры на 10–20 WU над землёй с учётом сжатия высот `/40` часто может быть уже достаточно.

### Вариант 2: Децимация массива высот (каждый N-й пиксель + улучшенный фильтр)
**Плюсы:** качество при том же количестве вершин значительно лучше nearest.  
**Минусы:** текущий код уже делает грубый даунсэмпл.  
**Оценка:** Обязательное дополнение к варианту 1 (добавить bilinear sampling).

### Вариант 3: LOD по расстоянию до камеры (near = детально, far = грубо)
**Плюсы:** хороший средний выигрыш. Легко посчитать дистанцию через `offsetX/Z` тайлов относительно `aircraftPosition`.  
**Минусы:** риск швов и popping.  
**Оценка:** Полезно, но **промежуточное решение**, если мы планируем радикальный переход на displacement map.

### Основной архитектурный вариант: Displacement Map

**Идея:** 
- Базовая `PlaneGeometry` с низким числом сегментов (пример: 16×16).
- Высоты упаковываются в текстуру (`DataTexture` float или RGBA-кодирование) из существующих `Float32Array`.
- Материал использует `displacementMap`.

**Точные цифры примера:**
- Базовая `PlaneGeometry(16, 16)` → **512 треугольников** на тайл (в 64 раза меньше текущих 32 768).

**Формулы (с учётом существующего кода):**
- `displacementScale = (data.maxElevation - data.minElevation) / 40`
- `displacementBias = - (data.minElevation - globalMinElev) / 40` (или нормализация 0..1 в текстуре)

**Плюсы:**
- Количество вершин падает в десятки раз (GPU тесселирует по текстуре).
- Визуальная детальность на средних/дальних дистанциях может вырасти.
- Отлично подходит под типичную высоту камеры 10–40 WU.
- Уже есть decoded heights (`min/maxElevation`) и спутниковые битмапы.

**Минусы:** небольшая переработка генерации мешей + материалов; нужно аккуратно работать с нормалями и швами.

**Оценка:** Самый перспективный долгосрочный подход для этой сцены.

## 3. Рекомендуемая стратегия (после обсуждения)

**Уточнённый подход (утверждён):**

1. **P0 (быстрый выигрыш)** — только снижение сегментов + улучшение семплинга.
   - Цель: быстро получить 6–8× снижение трисов и замерить реальный эффект на chase-камеру.

2. **Замер и решение** — после P0 оцениваем качество и FPS.

3. **P1 (основной скачок)** — сразу переход на **Displacement Map**, **пропуская тяжёлый ручной бакетный LOD** (бывший P1).
   - Ручной distance-based LOD с множеством бакетов делается **только если после P0 + displacement где-то всё равно будет тесно** (отложенный опциональный шаг).

**Принципы:**
- Максимально использовать уже существующие данные (`minElevation`, `maxElevation`, `globalMinElev`, `aircraftPosition`).
- Учитывать два режима (`realistic` / `schematic`) — в schematic можно быть гораздо агрессивнее по снижению.
- Все изменения должны быть измеримыми (трисы, FPS, визуальное качество).
- Избегать промежуточных сложных систем (ручной LOD), если конечная цель — displacement.

## 4. Поэтапный план реализации (обновлён)

### P0 — Быстрые победы + замер (0.5–1 день)

Цель: получить значительное снижение трисов с минимальными изменениями и замерить эффект.

- [ ] Снизить дефолтные сегменты (рекомендуется 32×64 как стартовая точка; можно 32×128 как более консервативный вариант).
- [ ] Улучшить семплинг высот до bilinear (добавить `sampleHeightBilinear` в `terrainTileUtils.ts`).
- [ ] В schematic-режиме (`mode === 'schematic'`) использовать значительно более низкие сегменты всегда (например 16×32 или 8×16).
- [ ] Убрать/сделать условным `frustumCulled = false` на terrain-мешах (или оставить только для ближних).
- [ ] Добавить временное логирование:
  - `trianglesPerTile`, `totalTerrainTriangles` при обновлении сетки.
  - Вывод в консоль / простой overlay.
- [ ] Провести замеры на "худшем случае":
  - Chase view над горами (реальные координаты, например Альпы).
  - Top view + зум.
  - Движение с включённым real terrain.

**Ожидаемый результат P0:**
- ~8× снижение (4 096 трисов на тайл вместо 32 768).
- При 25 тайлах: ~100–150k треугольников на terrain (вместо 800k+).
- Для chase на 10–20 WU — часто уже приемлемое качество.

### P1 — Displacement Map (основной качественный и количественный скачок, 2–4 дня)

**Технические детали реализации:**

- Генерация height-текстуры из `Float32Array` (в `TerrainManager` или отдельной утилите при подготовке `TerrainTileData`):
  - Предпочтительно: `THREE.DataTexture` с `FloatType` (1 или 2 канала).
  - Альтернатива: RGBA-кодирование (аналогично оригинальному Terrain-RGB) или нормализованные 0–1 значения.
  - Разрешение текстуры можно снижать (128×128 / 256×256 на тайл).

- Переработка `RealTerrainMesh`:
  - Вместо ручного построения `BufferGeometry` с высоким числом вершин — `new THREE.PlaneGeometry(baseSegments, baseSegments)`.
  - Пример: `PlaneGeometry(16, 16)` → **512 треугольников**.
  - Материал:
    ```ts
    const mat = new THREE.MeshStandardMaterial({
      map: satelliteTexture,
      displacementMap: heightTexture,
      displacementScale: (data.maxElevation - data.minElevation) / 40,
      displacementBias: - (data.minElevation - globalMinElev) / 40,
      ...
    });
    ```
  - Использовать существующий `globalMinElev` для согласованности всей сетки.

- Поддержка режимов:
  - `realistic`: полная displacement + спутниковая текстура.
  - `schematic`: можно отключать displacement или использовать очень низкое `baseSegments` + wireframe.

- Нормали:
  - `computeVertexNormals()` на низком базовом меше (дёшево).
  - Или позже — генерация нормалей из displacement в кастомном шейдере.

- Дополнительно:
  - Кэширование height-текстур (аналогично IndexedDB + серверному кэшу).
  - Оптимизация памяти (не держать одновременно полный Float32Array и текстуру, если возможно).
  - Передача `globalMinElev` и per-tile min/max в компонент.

**Цель P1:** снизить количество треугольников terrain до уровня **сотен–нескольких тысяч** на тайл (вместо десятков тысяч) при сохранении или улучшении визуального качества на рабочих высотах камеры.

### P2 — Дополнительные улучшения и опциональные шаги (по результатам замеров)

- [ ] Если после P0 + P1 на chase всё ещё хочется больше детализации в радиусе 30–60 WU — добавить лёгкий per-tile LOD (2–3 уровня сегментов для базовой PlaneGeometry при displacement).
- [ ] Объединение соседних тайлов одинакового уровня в более крупные меши (снижение draw calls).
- [ ] Кэширование сгенерированных height-текстур и геометрий.
- [ ] Автоматический выбор базового разрешения по профилю устройства.
- [ ] Простые skirts или edge-matching для устранения швов (если они проявятся).
- [ ] Интеграция статистики трисов в `TerrainLogPanel` или LatencyMonitor.

## 5. Риски и открытые вопросы (обновлено)

- **Швы между тайлами** — главный риск при displacement (разные `min/max` на соседних тайлах). `globalMinElev` уже помогает; может потребоваться `displacementBias` на уровне всей сетки или простые skirts.
- **Нормали и освещение** при сильном displacement на низком базовом меше.
- **Schematic / wireframe режим** — displacement в wireframe может выглядеть странно (нужен отдельный путь).
- **Chase view на очень малой высоте** — может потребовать более плотной базовой геометрии (24×24 вместо 16×16) для ближних тайлов.
- **Память** — несколько height-текстур (25–49 тайлов). Нужно тестировать.
- **Формат текстуры** — точность float vs encoded RGBA (протестировать на целевых устройствах).
- Открытый вопрос: достаточно ли будет P0 для большинства сценариев использования, или displacement нужен в любом случае?

## 6. Метрики успеха

**После P0:**
- Среднее количество треугольников terrain в chase-режиме ~100–200k (при 25 тайлах).
- Визуальное качество приемлемо на расстоянии 10–25 WU (chase).

**После P1 (displacement):**
- Среднее количество треугольников terrain < 30–50k (в зависимости от количества тайлов и базовых сегментов).
- FPS на integrated GPU при движении с real terrain стабильно ≥ 45–60.
- Отсутствие заметных блочных артефактов в chase на рабочих дистанциях.
- Сохранение работоспособности и читаемости schematic-режима.
- Отсутствие критичных швов и артефактов освещения.

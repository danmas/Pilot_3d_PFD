## 1. Краткое резюме

Проект представляет собой высококачественный специализированный инструмент для визуализации и отладки авиационной телеметрии в реальном времени. Сочетает:
- UDP-bridge от tnparserrt (132 поля)
- Полноценный Primary Flight Display (PFD)
- Продвинутую 3D-модель самолёта (Three.js + R3F)
- Встроенный Flight Dynamics Model (FDM) симулятор
- Конфигурируемый Panel Builder (drag-and-drop)
- Систему захвата/реплея (`.pfdrec`, blackbox)
- Raw Data Monitor и диагностический viewer

**Общая оценка инженерного качества: высокая (8.5–9/10 для нишевого инструмента).**

Проект выгодно отличается глубокой проработкой предметной области (ARINC, tnparserrt, физика полёта), отличной самодокументированностью и цельной архитектурой данных (`telemetry-frame.v1` + `field-catalog`).

## 3. Наблюдения и ограничения (технический долг / риски)

| Область                          | Текущее состояние | Влияние | Приоритет |
|----------------------------------|-------------------|---------|-----------|
| **Тестирование**                 | Только typecheck (`tsc --noEmit`). Тесты есть только в `packages/realtime-charts` (vitest для ring-buffer/decimator). Нет тестов decoding, simulator FDM, capture, bridge. | Высокий риск регрессий в самом сложном коде (бинарный протокол, физика). | P0 |
| **Монолитность glue-слоя**       | `bridge-plugin.ts` (~66 КБ) содержит UDP, simulator, SSE, capture, blackbox, все API-роуты, интеграцию с Vite. | Сложно тестировать/рефакторить/понимать в изоляции. Главный риск при развитии. | P0 |
| **Продакшен и развёртывание**    | Отдельный `server/server.js` (Express + встроенный terrain). Vite в dev. Много предупреждений про MIME, pm2, DISABLE_HMR. Terrain требует отдельного порта 3409 + токена + квоты. | Хрупкость деплоя (особенно с pm2). | P0 |
| **packages/realtime-charts**     | Отдельный пакет с собственным `node_modules`, vitest, планами (stacked/overlay). Не интегрирован в основной UI (есть только импорт ChartsView в App.tsx). | Раздутие репозитория, дубли node_modules, неясный статус. | P1 |
| **Зависимости и окружение**      | `@google/genai` присутствует + `GEMINI_API_KEY`, но активного использования почти нет (только в metadata и define). Mapbox токен обязателен для terrain. | Лишняя зависимость + риск утечки ключей. | P1 |
| **Отсутствие CI / автоматизации**| Нет упоминаний GitHub Actions / pipeline. Сборка, типы и тесты (когда появятся) выполняются вручную. | Риск сломанного main. | P1 |
| **Синхронизация с внешним парсером** | Полная зависимость от актуальности `out.json` (и `tn3.tcfg`). При изменении слотов — ручная синхронизация. Есть warnings, но нет автоматической миграции схемы. | Главная внешняя уязвимость. | P1 |
| **Дублирование в инструментах**  | Есть PFD (5 приборов) + отдельные *Instrument.tsx + PFD2. | Небольшая фрагментация. | P2 |
| **Артефакты в дереве**           | `node_modules` (включая вложенный), `dist/`, `captures/` (180+ файлов), `tmp/`, `.hermes/plans/`. | Загрязнение, размер репо. | P2 |
| **Документация продакшена**      | Хорошие инструкции, но неполное покрытие полного стека (terrain proxy + quota + Mapbox в одном месте). | Операционные риски. | P2 |

Другие замечания:
- `vite.config.ts`: `minify: false` в build (намеренно), сложные условия по `DISABLE_HMR`.
- `telemetryRef` + `aircraftControlsRef` — мощный, но неочевидный паттерн (требует дисциплины с хуками).
- Реальный terrain реализован недавно и качественно (кэш, quota, восстановление).

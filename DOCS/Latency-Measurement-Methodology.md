# Методика измерения задержки отображения (общий вид)

Документ описывает **архитектурно-нейтральную** методику проверки требования:

> Задержка от момента приёма данных сетевым интерфейсом до момента их визуализации не должна превышать **100 мс**.

Используется для сравнения разных реализаций (desktop Qt, Node.js + браузер и т.д.) на **одинаковых определениях метрик и сценариях**.

Конкретная реализация в Pilot_qt: [Latency-Measurement.md](Latency-Measurement.md).

---

## 1. Что измеряем

### 1.1. Основная метрика — display latency

**display_ms = T₁ − T₀**

| Точка | Определение | Примечание |
|-------|-------------|------------|
| **T₀** | Момент, когда **приложение-приёмник** прочитало UDP-datagram из сокета | Практичная софтверная точка; не требует доступа к NIC |
| **T₁** | Момент, когда **значение кадра отражено на экране** — завершена отрисовка UI-элемента, показывающего данные | Софтверный surrogate «визуализации» |

**Критерий (рекомендуемый):** P99(display_ms) ≤ 100 мс на типичном сценарии.

### 1.2. Вспомогательные метрики

Разбиение цепочки помогает локализовать узкое место при сравнении стеков:

| Метрика | Формула | Смысл |
|---------|---------|-------|
| **processing_ms** | T_decode − T₀ | Парсинг протокола, запись в модель данных |
| **queue_ms** | T_ui_ready − T_decode | Очереди потоков / event loop до готовности данных для UI |
| **render_ms** | T₁ − T_render_start | Время отрисовки (layout, paint, compositing) |
| **display_ms** | T₁ − T₀ | **Итог для п. 4.2** |

Не обязательно измерять все компоненты в первой итерации — достаточно **T₀, T_decode, T₁** и frame_id.

### 1.3. Идентификация кадра

Каждый UDP-фрейм (или логический кадр после сборки INFO+DATA) получает монотонный **frame_id**:

- счётчик протокола (например, `Counter` в INFO TNParserRT);
- или внутренний revision после decode.

Задержка считается **per frame**: «когда кадр N, принятый в T₀(N), впервые отображён в T₁(N)».

Если UI пропускает промежуточные кадры (coalesce, throttle), фиксируется revision **фактически отрисованного** кадра — иначе метрика не отражает то, что видит оператор.

## 2. Границы методики (что не входит)

| Включено (софтверно) | Не включено (опционально) |
|----------------------|---------------------------|
| Чтение из UDP-сокета процессом-приёмником | Timestamp на NIC / в ядре (SO_TIMESTAMPING) |
| Завершение paint/layout в UI | Фактический photon on screen (камера) |
| Статистика P50/P95/P99/MAX за сессию | Задержка TNParserRT и сети **до** UDP-порта клиента |

Для сравнения двух клиентов (Qt vs браузер) **достаточно одинаковых T₀ и T₁** — оба стека измеряются одинаково «слева» (read socket) и «справа» (paint complete).

Дополнительная погрешность T₁: compositor и vsync монитора (**~0–16 мс**) — одинаково для обоих desktop/browser и обычно не меняет вывод при пороге 100 мс.

## 3. Процедура испытания

### 3.1. Стенд

```
Источник (TNParserRT / эмулятор)
        │ UDP
        ▼
   Клиент A или B  ──►  UI (desktop / browser)
        │
        └── лог метрик (CSV / overlay)
```

**Фиксируются в протоколе:**

- версия клиента;
- ОС, CPU, монитор;
- `out.json`, порт UDP, `tickcount`;
- число отображаемых параметров / виджетов;
- режим UI (приборная панель, графики, таблица).

### 3.2. Сценарии

| ID | Частота источника | Нагрузка UI | Цель |
|----|-------------------|-------------|------|
| **S1** | tickcount=40 (~25 Hz) | типичный экран полёта | номинальный режим |
| **S2** | tickcount=1 (~1024 Hz) | тот же экран | worst case по потоку |
| **S3** | tickcount=40 | максимум параметров на экране | worst case по UI |
| **S4** | S1 + фоновая нагрузка CPU ~50% | устойчивость | |

**Длительность:** ≥ 5 мин на сценарий (при 25 Hz ≈ 7500 кадров).

### 3.3. Сбор данных

На каждый отрисованный кадр:

```
frame_id, t_recv_ms, t_decoded_ms, t_paint_ms, processing_ms, display_ms, view_mode
```

Хранить последние N записей (например, 5000) + агрегаты в реальном времени.

### 3.4. Обработка

1. Построить гистограмму display_ms.
2. Вычислить **P50, P95, P99, MAX**.
3. Посчитать **count(display_ms > 100)** и долю %.
4. Сравнить сценарии S1–S4.

**Прохождение (рекомендация):**

| Метрика | S1 (номинал) |
|---------|--------------|
| P99 display_ms | ≤ 100 мс |
| MAX display_ms | ≤ 150 мс (запас, согласовать в ТЗ) |
| Доля > 100 мс | → 0 |

## 4. Сопоставление стеков: Qt vs Node.js + браузер

Одинаковые определения T₀/T₁ позволяют сравнить цепочки.

### 4.1. Цепочки обработки

**Pilot_qt (desktop):**

```
T₀  QUdpSocket::readDatagram()
    → decode TNParserRT (фоновый поток)
    → ingest в RealtimeDataHub
    → UI thread: display tick / paintEvent
T₁  завершение paintEvent виджета
```

**Node.js + браузер (типовая схема):**

```
T₀  dgram/socket.on('message')  — Node.js прочитал UDP
    → parse в Node
    → передача в браузер (WebSocket / SSE / HTTP chunk)
    → JS: обновление state / DOM / Canvas
    → requestAnimationFrame → draw
T₁  завершение rAF-callback после записи в canvas/DOM
    (или PerformanceObserver: 'paint' / element update)
```

### 4.2. Где ставить метки (mapping)

| Этап | Pilot_qt | Node.js + browser |
|------|----------|-------------------|
| **T₀** | `readDatagram` + epoch ms | `socket.on('message')` + `Date.now()` / `performance.timeOrigin` |
| **T_decode** | после `ingestFrame` | после parse + перед send WS |
| **T_bridge** | — (данные in-process) | WS `send` → browser `onmessage` |
| **T_ui_ready** | после `chartDataChanged` | после `setState` / store update |
| **T_render_start** | начало `refreshActiveView` | начало rAF / `draw()` |
| **T₁** | конец `paintEvent` | конец rAF после canvas fill / DOM text update |

### 4.3. Ожидаемые различия (гипотезы для сравнения)

| Фактор | Qt desktop | Browser |
|--------|------------|---------|
| Передача данных UI | in-process, QueuedConnection | +1 hop (WS), сериализация JSON/binary |
| Event loop | 50 ms display tick | rAF ~16 ms, но + bridge latency |
| Отрисовка графиков | QPainter, coalesce 50 ms | Canvas/WebGL, свой throttle |
| Поток UDP | отдельный QThread | libuv thread pool → main thread Node → browser |
| Типичный риск | display tick, coalesce | WS jitter, GC, layout thrashing |

Сравнение **корректно**, если:

- один и тот же UDP-поток и `out.json`;
- одинаковое число параметров на экране;
- **T₀** в обоих случаях — read socket **клиентского процесса** (Node для browser-стека; Qt для desktop);
- **T₁** — момент обновления **конкретного видимого элемента**, а не «данные лежат в памяти».

### 4.4. Что включать в display_ms для browser-стека

Два варианта — **выбрать один и зафиксировать в протоколе**:

| Вариант | T₀ | T₁ | Интерпретация |
|---------|----|----|---------------|
| **A (end-to-end клиента)** | Node `on('message')` | browser rAF paint | Полный путь «UDP → экран» через два процесса |
| **B (только браузер)** | browser `onmessage` | rAF paint | Latency UI; не включает Node parse + WS |

Для **прямого сравнения с Qt** используйте **вариант A** — аналог «от read до paint» всего клиентского решения.

## 5. Минимальный набор для реализации в любом стеке

1. **Счётчик кадров** (frame_id) на каждый декодированный фрейм.
2. **T₀** при первом read UDP в процессе-приёмнике.
3. **T_decode** после parse.
4. **T₁** при первом paint, где frame_id отображён пользователю.
5. **Ring buffer** последних N замеров + **P50/P95/P99/MAX** online.
6. **CSV export** для протокола испытаний.
7. **Reset** статистики перед каждым сценарием.

## 6. Шаблон протокола испытаний

```
Дата:
Клиент: [Pilot_qt | Node+Browser v…]
Сценарий: S1 / S2 / S3 / S4
Источник: TNParserRT, tickcount=…, порт=…
Экран: N параметров, режим [панель | графики]

Результаты (display_ms):
  N samples: …
  P50: … ms
  P95: … ms
  P99: … ms
  MAX: … ms
  >100 ms: … (…%)

processing_ms P99: … ms   (опционально)
bridge_ms P99: … ms       (только browser-стек)

Вывод: [PASS | FAIL] по P99 ≤ 100 ms
```
